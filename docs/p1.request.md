# P1 需求分析

## 1. 需求概述

### 1.1 空闲超时自动踢出

**需求描述：** 用户连接桌面后，如果长时间（如 5 分钟）没有任何输入操作（键盘、鼠标、手柄等），自动断开该用户的连接并冻结会话，释放桌面资源。

**原因：** 当前桌面一次只能连接一个用户，空闲用户长期占用桌面会导致其他用户无法使用，造成资源浪费。

---

### 1.2 二次访问会话冲突处理

**需求描述：** 当新用户访问桌面时，如果已有用户正在占用（前一个连接未退出），需要：

1. 给新用户明确的提示，告知当前桌面已被占用
2. 提供操作选项，允许新用户强制踢掉前一个用户并接管桌面

**原因：** 当前二次访问时没有任何提示，也没有踢掉前一个用户的机制，新用户只能等待或无法使用。

---

## 2. 需求分析

### 2.1 空闲超时自动踢出

**涉及模块：**

- 前端 `addons/gstreamer-web/src/input.js` — 用户输入事件的源头，需要在此追踪最后活跃时间
- 前端 `addons/gstreamer-web/src/app.js` — Vue 应用主逻辑，需要增加空闲检测定时器和超时断连逻辑
- 前端 `addons/gstreamer-web/src/webrtc.js` — WebRTC 连接管理，提供 `sendDataChannelMessage` 和 `reset()` 用于通知服务端和断开连接
- 后端 `src/selkies_gstreamer/signalling_web.py` — 信令服务器，可选的服务端空闲检测（双保险）

**实现思路：**

1. 在前端 `Input` 类中记录 `lastActivityTime`，每次键盘/鼠标/手柄事件触发时更新
2. 在 `app.js` 中启动定时器（如每 30 秒检查一次），判断 `Date.now() - lastActivityTime` 是否超过阈值（默认 5 分钟，可配置）
3. 超时后：
   - 通过 data channel 通知服务端用户空闲（如发送 `idle_timeout` 消息）
   - 调用 `signalling.disconnect()` 断开信令连接
   - 前端显示"因长时间无操作，连接已断开"的提示，提供重新连接按钮
4. 超时时间应可通过环境变量或启动参数配置（如 `SELKIES_IDLE_TIMEOUT=300`）

**关键点：**

- 当前 `Input` 类已经处理了所有用户输入事件（`_mouseButtonMovement`、`_key`、`_touch`、`_gamepadButton` 等），只需在这些方法中统一更新活跃时间戳
- 服务端 `signalling_web.py` 的 `recv_msg_ping` 已有 keepalive 机制，可在此基础上扩展空闲检测作为双保险
- 需要注意：仅检测用户主动输入，不将 WebRTC 内部的 ping/pong、stats 上报等自动消息计为活跃

---

### 2.2 二次访问会话冲突处理

**涉及模块：**

- 后端 `src/selkies_gstreamer/signalling_web.py` — 信令服务器的 `hello_peer` 和 `connection_handler`，需要检测 peer_id 冲突并处理
- 前端 `addons/gstreamer-web/src/signalling.js` — 信令客户端，需要处理新的冲突消息类型
- 前端 `addons/gstreamer-web/src/app.js` — 需要增加冲突提示 UI 和强制接管操作

**当前问题：**

- `signalling_web.py` 的 `hello_peer` 中，如果 `uid in self.peers`（peer_id 已存在），直接关闭连接并抛异常，新用户无法连入
- 前端使用固定的 `peer_id = 1`，所有用户都用同一个 ID，必然冲突

**实现思路：**

1. 修改后端 `hello_peer`：当检测到 `uid` 已存在时，不直接拒绝，而是向新连接发送 `PEER_BUSY` 消息（携带已有用户信息）
2. 前端 `signalling.js` 的 `_onServerMessage` 增加对 `PEER_BUSY` 消息的处理，触发新的回调 `onpeerbusyy`
3. 前端 `app.js` 显示冲突提示对话框：
   - 提示文案："当前桌面已被其他用户占用"
   - 提供"强制接管"按钮
4. 用户点击"强制接管"后：
   - 前端发送 `KICK_PEER` 消息到信令服务器
   - 后端收到后调用 `remove_peer` 踢掉旧连接
   - 向旧用户发送断开通知
   - 新用户自动重新连接并建立会话

**关键点：**

- 当前 `self.peers` 字典以 `uid` 为 key 管理所有连接，`self.sessions` 管理会话配对关系，踢人逻辑需要同时清理这两个数据结构
- `cleanup_session` 和 `remove_peer` 方法已经实现了完整的清理逻辑，可以直接复用
- 被踢用户的前端会触发 `_onServerClose` → `ondisconnect` 回调链，需要在此场景下显示"您已被其他用户踢出"的提示，而非默认的重连逻辑
